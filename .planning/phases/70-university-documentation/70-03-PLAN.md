1. **Einleitung** (3-5 Saetze): Zweck des Dokuments, Kurzueberblick der App

2. **Architektur** (per D-14):

   2.1 **Ueberblick**: Die App folgt einer 3-Schicht-Architektur:
   - **Praesentationsschicht**: React Server/Client Components, shadcn/ui, Tailwind CSS
   - **Logikschicht**: Next.js Server Actions, Zod-Validierung, Exercise Engine (pure functions)
   - **Datenschicht**: Supabase (PostgreSQL), Row Level Security, SECURITY DEFINER Helfer

   2.2 **Next.js App Router**: Erklaerung des Route-Group-Patterns:
   - `(child)` Route Group fuer alle Kind-Seiten
   - `(teacher)` Route Group fuer Lehrer-Dashboard
   - Middleware prueft JWT-Claims und leitet auf rollenbasierte Routen um
   - Server Components laden Daten, Client Components bieten Interaktivitaet

   2.3 **Supabase-Integration**:
   - Drei Client-Typen: Browser-Client (lib/supabase/client.ts), Server-Client (lib/supabase/server.ts), Admin-Client (lib/supabase/admin.ts)
   - Server-Client fuer Server Components und Server Actions (Cookie-basierte Auth)
   - Admin-Client mit Service-Role-Key fuer privilegierte Operationen (Kind anlegen)
   - Browser-Client fuer clientseitige Auth-Zustandspruefung

   2.4 **Sicherheitskonzept**:
   - Row Level Security (RLS) auf allen 4 Tabellen
   - SECURITY DEFINER Helfer-Funktionen in privatem Schema (private.user_role(), private.user_class_id(), private.user_school_id())
   - Kinder sehen nur eigene Daten, Lehrer sehen Daten ihrer Klasse
   - PIN-basierte Auth fuer Kinder: PIN wird als Passwort in auth.users gespeichert (bcrypt-Hash), kein Klartext
   - Stateless Answer Validation: Server re-berechnet korrekte Antwort, vertraut nicht dem Client

3. **Tech-Stack-Entscheidungen** (per D-15):
   Tabelle mit folgenden Eintraegen:

   | Technologie | Version | Begruendung |
   |-------------|---------|-------------|
   | Next.js | 15.2 | App Router, Server Components, Server Actions — alles in einem Framework |
   | React | 19 | Neueste Version, Server Component Support |
   | Supabase | 2.103 | PostgreSQL + Auth + RLS out-of-the-box, kein eigener Auth-Server noetig |
   | TypeScript | 5.x | Typsicherheit, bessere IDE-Unterstuetzung, weniger Laufzeitfehler |
   | Tailwind CSS | 4.x | Utility-first CSS, schnelles Prototyping, kindgerechtes Design |
   | shadcn/ui | — | Barrierefreie, anpassbare UI-Komponenten (Button, Card, Table, etc.) |
   | Zod | 4.x | Schema-Validierung fuer Server Actions, typsicher |
   | Vitest | 2.x | Schnelle Unit-Tests, kompatibel mit Vite-Ecosystem |
   | Playwright | 1.59 | E2E-Tests im echten Browser, zuverlaessig |

   Unter der Tabelle: Begruendung gegen Alternativen:
   - Warum Supabase statt Firebase (PostgreSQL + RLS vs. NoSQL + Firestore Rules)
   - Warum Server Actions statt REST-API (weniger Boilerplate, typsicher, keine API-Routes noetig)
   - Warum CSS-Animationen statt Phaser.js fuer Mini-Game (geringere Bundle-Groesse, kein Game-Engine-Overhead)

4. **Implementierungsschritte** (per D-16):

   Fuer jede Phase einen Unterabschnitt mit:
   - Was wurde gebaut (2-3 Saetze)
   - Welche Dateien/Module wurden erstellt
   - Welche technischen Entscheidungen wurden getroffen

   4.1 **Phase 10: Foundation**
   - Next.js 15.2 Projekt mit TypeScript, Tailwind CSS 4, shadcn/ui (New York Style)
   - Supabase Cloud-Projekt in EU/Frankfurt, PostgreSQL-Schema mit 4 Tabellen
   - Dual-Login: Kinder via Benutzername+PIN (4-stellig, padded zu Email), Lehrer via Email+Passwort
   - Middleware routet basierend auf JWT-Claims zu /kind/* oder /lehrer/*
   - RLS-Policies mit SECURITY DEFINER Helfern in privatem Schema
   - Trigger erstellt automatisch Lehrer-Profil bei Registrierung

   4.2 **Phase 20: Exercise Engine**
   - Pure-Function-Engine ohne Datenbankabhaengigkeit
   - RANGES-Konfiguration: 4 Klassenstufen x 3 Schwierigkeitsgrade = 12 Kombinationen
   - Curriculum-Constraints: Klasse 1 Addition/Subtraktion bis 20, Klasse 2 bis 100, Klasse 3 Einmaleins, Klasse 4 gemischt
   - Schwierigkeits-Adaption: Aufstieg nach 3x richtig, Abstieg nach 2x falsch
   - Stateless Validation: Server berechnet korrekte Antwort neu (vertraut nicht dem Client)

   4.3 **Phase 30: Kind-Dashboard und Uebungssession**
   - Dashboard zeigt Punkte, Fortschrittsbalken zum Mini-Game, Klassenstufe
   - NumberPad: Touch-freundliches 3x4-Grid fuer Kinder (keine Tastatur noetig)
   - State Machine: loading -> answering -> submitting -> feedback -> naechste Aufgabe
   - Visuelles Feedback: Gruen (richtig + Punkte), Rot (falsch + korrekte Antwort)

   4.4 **Phase 40: Mini-Game (Ballonplatzen)**
   - CSS-basiertes Ballon-Spiel statt Game-Engine (kein Phaser.js/Canvas noetig)
   - Ballons steigen auf, Kind klickt zum Platzen, 75 Sekunden Timer
   - Punkte-Gate: 100 Punkte erforderlich, werden beim Spielstart abgezogen
   - Rein clientseitiger Score (kein Schreiben in DB, per Design)

   4.5 **Phase 50: Lehrer-Dashboard**
   - Server Component laedt Klassendaten ueber Supabase-Queries
   - Sortierbare Tabelle mit Schueler-Uebersicht (Punkte, Aufgaben, Genauigkeit, letzte Aktivitaet)
   - Expandierbare Zeilen zeigen Genauigkeit pro Operationstyp
   - Inaktive Schueler (>7 Tage) farblich hervorgehoben
   - Alle Daten durch RLS auf eigene Klasse beschraenkt

   4.6 **Phase 60: Abo-Gate (Subscription)**
   - Klasse 1-3: kostenlos, kein Gate
   - Klasse 4 + Free-Tier: Redirect auf Upgrade-Seite
   - Simulierter Checkout: 3 Pakete (Grundschulniveau 9,99 EUR, Foerderung 14,99 EUR, Experte 19,99 EUR)
   - Demo-Tier fuer Evaluierung (kein Gate bei demo)
   - Lehrer-Dashboard zeigt Abo-Status als Badge
