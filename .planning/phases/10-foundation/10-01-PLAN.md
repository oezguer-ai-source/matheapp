## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Developer machine -> Git | Secrets must not cross this boundary |
| Supabase cloud dashboard -> Developer machine | Service-role key travels once during setup; must land in gitignored file |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01-01 | Information Disclosure | `.env.local` committed to git | mitigate | `.gitignore` contains `.env.local` AND `.env*.local` BEFORE the checkpoint runs. Checkpoint instructs dev to verify `git status` does not list `.env.local`. |
| T-10-01-02 | Elevation of Privilege | `SUPABASE_SERVICE_ROLE_KEY` exposed to browser | mitigate | `.env.example` template uses the non-`NEXT_PUBLIC_` prefix exclusively for service-role. Downstream plans enforce this boundary in `lib/supabase/admin.ts` (Plan 04). |
| T-10-01-03 | Spoofing | Weak database password | accept | Supabase enforces its own password requirements during project creation; developer saves strong generated password to password manager. Low-value target for an unreleased app. |
| T-10-01-04 | Information Disclosure | `SUPABASE_ACCESS_TOKEN` leaked | mitigate | Token scoped to single developer account, revocable at `Account -> Access Tokens`. Stored only in gitignored `.env.local`. |
</threat_model>
